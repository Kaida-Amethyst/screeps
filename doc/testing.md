# 测试与验证策略

本文档记录正式版 Screeps Arena binding 在当前阶段可行的测试与验证策略。

## 当前现实约束

当前无法在本地稳定获得完整的 Screeps Arena 运行时环境，因此：

- 不能把正式版 binding 的测试目标设定为“非常认真的自动化运行时测试”
- 不能像普通纯 MoonBit 库那样依赖完整单元测试覆盖
- 不能假设每个 API 都能在本地通过真实运行时自动回归验证

因此当前必须接受一个现实：

- 我们只能做比较 naive 的测试与验证

## 当前结论

正式版第一阶段的验证策略应明确定位为：

- 静态验证
- 编译验证
- 少量结构检查
- 手工 smoke test

而不是：

- 全自动的高置信度运行时测试体系

## 当前可做的 naive 验证

### 1. 静态与编译层验证

每次重要改动后，至少运行：

- `moon check`
- `moon info`
- `moon fmt`
- `moon test`
- `moon run main --target js --build-only`

这一层用于确保：

- MoonBit 类型正确
- 包接口没有意外漂移
- 代码格式正常
- JS 产物可以构建

### 2. 生成产物检查

当前正式流程是：

- `main.mjs`
- `moonbit-main.mjs`

因此可以做一些轻量检查，例如：

- MoonBit 产物仍然导出预期入口
- wrapper 仍然导入 `moonbit-main.mjs`
- wrapper 需要的 helper 名仍然存在

这类检查不验证游戏行为，但可以防止集成链路悄悄断裂。

### 3. 手工 smoke test

当前必须接受手工验证的存在。

可行流程包括：

- 将生成产物拷贝到 `~/ScreepsArena/tutorial/`
- 点击 `Play`
- 观察是否能正常运行
- 用少量代表性场景做回归

当前最重要的手工 smoke test 场景包括：

- tutorial 主线
- final test
- 至少一个完整 bot

## 当前做不到的事情

在当前阶段，以下目标不现实：

- 完整自动化的 Screeps runtime 回归测试
- 高覆盖率的行为级单元测试
- 对所有 API 做严格的真实运行环境回归

这些限制不是工程疏忽，而是宿主环境约束决定的。

## 因此应如何设定“通过标准”

正式版第一阶段的“通过”标准应当现实一些：

- 编译通过
- 关键示例能构建
- wrapper 链路没有断
- tutorial 主线能手工跑通
- final test 能手工跑通

这类标准虽然朴素，但对当前 Screeps Arena binding 是合理的。

## 当前态度

- 不伪装成拥有完整严肃测试体系
- 明确承认当前测试方式偏 naive
- 通过文档把这一点说清楚，避免后续误判质量边界

## 后续可能的改进方向

如果未来有更稳定的本地运行环境或可脚本化入口，再讨论：

- 自动化 smoke test
- tutorial 回归脚本
- wrapper 集成测试
- 更严格的行为检查

但这些不属于当前第一阶段的前提条件。
